# Especificacao Operacional Para Agentes de IA - `15.05.01.csv`

## 1. Objetivo
Este documento define um contrato tecnico para agentes de IA criarem automacoes sobre o arquivo `15.05.01.csv` (relatorio de lancamentos OBZ do Promax).

Objetivos da automacao:
- ler e normalizar o CSV com seguranca;
- interpretar corretamente o campo `Documento`;
- reconciliar `Comprometido` e `Realizado`;
- preservar o `Historico` relevante em lancamentos `CP` e `NC`;
- gerar saidas consistentes para analise financeira e auditoria.

## 2. Escopo e Nao Escopo
Escopo:
- processamento de lancamentos OBZ no layout atual;
- conciliacao de eventos de pagamento;
- enriquecimento de historico.

Nao escopo:
- alteracao de regras contabeis do ERP;
- retificacao de dados de origem;
- inferencia de dados externos ao arquivo.

## 3. Contrato de Entrada

### 3.1 Formato do arquivo
- Tipo: CSV
- Delimitador: `;`
- Encoding: `latin1`
- Data esperada: `dd/mm/yyyy`
- Ordem original das linhas: deve ser preservada (criar `row_index` 1-based)

### 3.2 Cabecalho esperado
O arquivo deve conter exatamente estas colunas (nomes identicos):
- `Pacote`
- `NBZ`
- `Departamento`
- `Data`
- `Documento`
- `Fornecedor`
- `Nome Fornecedor`
- `Historico`
- `Conta Gerencial`
- `VBZ`
- `Comprometido`
- `Realizado`
- `Usuario`

Se faltar coluna obrigatoria, interromper processamento com erro `E001_MISSING_REQUIRED_COLUMN`.

### 3.3 Tipagem logica esperada
- Texto: todas as colunas, exceto campos monetarios e data apos parse.
- Data: `Data` -> tipo date.
- Monetario: `Comprometido` e `Realizado` -> decimal(18,2).

## 4. Normalizacao Canonica (Obrigatoria)
Todas as regras abaixo devem ser idempotentes.

### 4.1 Normalizacao de texto
Aplicar nas colunas textuais:
1. remover espacos nas pontas (`strip`);
2. colapsar espacos internos multiplos para um unico espaco (`\s+ -> " "`).

Observacao:
- manter conteudo semantico original (nao remover palavras relevantes).

### 4.2 Normalizacao de data
Regra:
- parse estrito `dd/mm/yyyy`.

Se falhar parse, registrar `E002_INVALID_DATE_FORMAT`.

### 4.3 Normalizacao monetaria
Entrada tipica:
- `+     11.176,00`
- `-        130,00`
- `+          0,00`

Algoritmo:
1. remover espacos;
2. capturar sinal (`+` ou `-`, padrao `+` se ausente);
3. remover separador de milhar (`.`);
4. trocar separador decimal `,` por `.`;
5. converter para decimal(18,2).

Se falhar conversao, registrar `E003_INVALID_MONEY_FORMAT`.

## 5. Parse da Coluna `Documento`

### 5.1 Normalizacao previa
- aplicar normalizacao de texto;
- usar `Documento_Normalizado`.

### 5.2 Tipos suportados
- `CP`: Contas a Pagar (`CP <titulo> <parcela>`)
- `NC`: Nota de Compra (`NC <nota> <serie> <parcela>`)
- `CB`: Caixa Banco (`CB <documento>`)
- `MN`: Manual (`MN <documento>`)
- `PC`: Pedido de Compra (`PC <pedido>`)

### 5.3 Regex oficiais
- `CP`: `^CP\\s+(?P<titulo>\\d+)\\s+(?P<parcela>\\d+)$`
- `NC`: `^NC\\s+(?P<nota>\\d+)\\s+(?P<serie>\\S+)\\s+(?P<parcela>\\d+)$`
- `CB`: `^CB\\s+(?P<documento>\\d+)$`
- `MN`: `^MN\\s+(?P<documento>\\d+)$`
- `PC`: `^PC\\s+(?P<pedido>\\d+)$`

Se nao casar com nenhum padrao, registrar `E004_UNKNOWN_DOCUMENT_PATTERN`.

### 5.4 Campos derivados obrigatorios
- `Tipo_Documento` (`CP`, `NC`, `CB`, `MN`, `PC`)
- `Documento_Normalizado`
- `Doc_Numero_Principal` (titulo/nota/documento/pedido)
- `Doc_Serie` (somente `NC`, senao vazio)
- `Doc_Parcela` (`CP`/`NC`, senao vazio)

## 6. Modelo de Evento Contabil

### 6.1 Classificacao de linha
Com base nos valores normalizados:
- `COMPROMETIDO_INICIAL`: `Comprometido > 0` e `Realizado == 0`
- `ESTORNO_COMPROMETIDO`: `Comprometido < 0` e `Realizado == 0`
- `REALIZADO`: `Realizado != 0`
- `NEUTRO`: demais casos

### 6.2 Semantica por tipo
- `CP` e `NC`: podem gerar ciclo completo:
1. `COMPROMETIDO_INICIAL`
2. `ESTORNO_COMPROMETIDO`
3. `REALIZADO`

- `CB`, `MN`, `PC`: podem ter apenas `REALIZADO` e nao exigem ciclo de estorno.

## 7. Chave de Conciliacao e Matching

### 7.1 Regra principal
`Documento` isolado nao e chave unica.

Usar chave composta base para reconciliar eventos:
- `Tipo_Documento`
- `Documento_Normalizado`
- `Fornecedor`
- `Conta Gerencial`
- `NBZ`
- `Departamento`
- `Pacote`

### 7.2 Regra de desambiguacao
Quando houver multiplas candidatas na mesma chave:
1. priorizar mesma magnitude de valor:
- `abs(Comprometido_Inicial) == abs(Estorno)` para estorno;
- `abs(Comprometido_Inicial) ~= abs(Realizado)` (tolerancia `0.01`) para pagamento.
2. priorizar `Data` mais proxima;
3. priorizar linha com `Data` menor ou igual ao `Realizado`;
4. desempate final por `row_index` crescente.

### 7.3 Estados de reconciliacao
- `PAGO`: possui linha `REALIZADO` conciliada.
- `ABERTO`: possui `COMPROMETIDO_INICIAL` sem `REALIZADO`.
- `REALIZADO_SEM_ORIGEM`: possui `REALIZADO` sem comprometido correspondente.
- `INCONSISTENTE`: conflito de matching nao resolvido.

## 8. Regra de Historico Para `CP` e `NC`

### 8.1 Problema funcional
Para `CP` e `NC`, o historico detalhado do usuario costuma existir na linha de `COMPROMETIDO_INICIAL`, enquanto a linha de `REALIZADO` costuma trazer apenas fornecedor.

### 8.2 Regra obrigatoria de heranca
Para cada linha `REALIZADO` de `CP`/`NC`:
1. localizar `COMPROMETIDO_INICIAL` conciliado;
2. definir:
- `Historico_Original` = historico original da linha atual;
- `Historico_Efetivo` = historico do `COMPROMETIDO_INICIAL` conciliado;
- `Historico_Origem` = `COMPROMETIDO`.

Fallback:
- se nao encontrar comprometido conciliado, manter:
- `Historico_Efetivo = Historico_Original`
- `Historico_Origem = REALIZADO`
- registrar `W003_REALIZADO_WITHOUT_COMMIT_MATCH`.

### 8.3 Regra para demais tipos (`CB`, `MN`, `PC`)
- `Historico_Efetivo = Historico_Original`
- `Historico_Origem = REALIZADO`

## 9. Contrato de Saida (Automacao)
Gerar duas saidas padrao.

### 9.1 Tabela 1 - `lancamentos_normalizados`
Uma linha por linha de entrada, com colunas:
- todas as colunas originais;
- `row_index`;
- `Tipo_Documento`;
- `Documento_Normalizado`;
- `Doc_Numero_Principal`;
- `Doc_Serie`;
- `Doc_Parcela`;
- `Data_Normalizada`;
- `Comprometido_Valor`;

- `Historico_Original`;
- `Historico_Efetivo`;
- `Historico_Origem`;
- `Event_Key`;
- `Status_Reconciliacao`;
- `Quality_Flags` (lista ou string com codigos).

### 9.2 Tabela 2 - `eventos_consolidados`
Uma linha por evento reconciliado, com colunas minimas:
- `Event_Key`
- `Tipo_Documento`
- `Documento_Normalizado`
- `Fornecedor`
- `Conta Gerencial`
- `NBZ`

## 11. Fluxo Obrigatorio Para Agente de IA
Executar nesta ordem:
1. validar contrato de entrada;
2. criar `row_index` e normalizar campos;
3. parsear `Documento`;
4. classificar linhas (`Classe_Linha`);
5. reconciliar eventos (`Event_Key`);
6. enriquecer historico (`Historico_Efetivo`);
7. gerar tabelas de saida;
8. emitir relatorio de qualidade com contagem por codigo `E*` e `W*`.

## 12. Pseudocodigo de Referencia
```text
load_csv(delimiter=';', encoding='latin1')
validate_required_columns()
add_row_index()

normalize_text_columns()
parse_date_column('Data')
parse_money_columns('Comprometido', 'Realizado')
parse_documento()

classify_rows()
build_candidate_groups(composite_key)
reconcile_commits_reversals_realizados()

for each row:
  if Tipo in (CP, NC) and Classe_Linha == REALIZADO:
    Historico_Efetivo = historico_do_comprometido_conciliado_ou_fallback()
  else:
    Historico_Efetivo = Historico_Original

write_lancamentos_normalizados()
write_eventos_consolidados()
write_quality_report()
```

## 13. Casos de Teste Minimos
O agente deve validar pelo menos estes cenarios:
1. `CP` com ciclo completo (`+Comprometido`, `-Comprometido`, `+Realizado`) e heranca de historico correta.
2. `NC` com `Realizado` diferente do `Comprometido` (pagamento parcial ou ajuste).
3. `CB` com apenas `Realizado`, sem tentativa de herdar historico.
4. dois eventos com mesmo `Documento`, mas `Fornecedor` diferente (sem cruzamento indevido).
5. linha com `Documento` invalido gerando `E004`.

## 14. Observacoes de Implementacao
- manter rastreabilidade: sempre preservar colunas originais.
- usar comparacao monetaria com tolerancia (`0.01`) para evitar erro de ponto flutuante.
- nao descartar linhas inconsistentes; marcar e seguir com log.
- toda decisao de fallback deve gerar flag em `Quality_Flags`.
